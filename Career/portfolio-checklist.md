# Portfolio Checklist

*Guidance: a portfolio's job is to make a reviewer believe you can do
the job, in the time they're actually willing to spend (usually under 2
minutes per candidate). Check against your real, live portfolio site/repo.*

---

## Overall structure

- [ ] The 2-3 best projects are surfaced first — not chronological order,
      not every project you've ever built
- [ ] Each project has a one-line summary visible before any click is
      required (what it does + who it's for)
- [ ] Contact/resume/LinkedIn links are visible without scrolling

## Per-project entry

Use `Technologies/Templates/portfolio-project.md` for the write-up itself.
Check each featured project against:
- [ ] States YOUR specific contribution clearly, especially for team
      projects — not just "we built X"
- [ ] Includes a live demo link or video — a portfolio piece with only a
      GitHub link asks the reviewer to do the work of understanding it
- [ ] States the key technical decision(s), not just the tech stack list
- [ ] Is honest about limitations/what you'd improve — this reads as
      more credible, not less impressive

## Repo hygiene (for linked GitHub projects)

- [ ] README exists and follows `Technologies/Templates/readme.md`
- [ ] Repo isn't full of commented-out code, TODO clutter, or an
      unexplained folder structure
- [ ] Commit history looks like real, incremental work — not one giant
      "final commit"
- [ ] It actually still runs if someone clones it today (dependencies
      pinned, no dead external service calls)

## Before calling it done

- [ ] Ask someone outside your immediate field to spend 2 minutes on
      your portfolio and say back what they think you can do — if it's
      unclear or wrong, the summary/ordering needs work.
- [ ] Check on mobile — many reviewers click through from LinkedIn/email
      on a phone first.
